คุณสมบัติอย่างหนึ่งของ Go คือ มันมี Garbage collector (GC) ทำให้ชีวิต Dev อย่างเรา ๆ ไม่ต้องมาปวดหัวกับการที่ต้องมาคอยจัดการหน่วยความจำ ซึ่งความสะดวกนี้มันก็มีสิ่งที่ต้องแลกมานั่นคือ ช่วงเวลาที่ต้องให้ GC ทำงานนั่นเอง ถึงจะไม่ขนาด stop-the-world ก็เถอะนะ [เพิ่มเติม Go GC ได้ตามลิ้งก์นี้เลย](https://tip.golang.org/doc/gc-guide) แต่สำหรับบางงานที่ latency คือสิ่งสำคัญเราก็ต้องมาช่วยให้ GC ทำงานน้อยลงนั่นคือลด heap allocation ลงนั่นเอง เป็นที่มาว่าทำยังไงเราจะลด heap allocation ให้ได้มาและไม่ฝืนจนเกินไป

<!--more-->

## ทำไมต้องหลีกเลี่ยง heap allocation?
แม้ว่า GC ของ Go จะถูกออกแบบมาเพื่อความมีประสิทธิภาพในการจัดการการหน่วยความจำได้อย่างรวดเร็วแต่ก็มีข้อสังเกตุอยู่:
- Latency: ทุก ๆ รอบของการทำงาน GC จะมีความหน่วงเล็กน้อยเสมอ ซึ่งอาจเป็นปัญหาในระบบที่ต้องการ response time ที่สม่ำเสมอมาก ๆ
- CPU resource: GC จะทำงานได้ก็ต้องใช้ CPU เพราะฉะนั้นจะมีการแบ่ง resource ไปใช้กับ GC
- ~~Stop-The-World~~: ถึงแม้ว่าจะไม่ถึงขั้น stop-the-world แบบจริงจังแต่มันก็จะมีช่วงที่ต้องหน่วงเล็กน้อยอยู่ดีซึ่งก็จะมีผลกับระบบที่ซับซ้อนหรือมีขนาดใหญ่ มาก ๆ อยู่ดี

ซึ่งเราสามารถช่วยลดภาระงานของ GC ลงได้ด้วยการ~~ไม่ทิ้งขยะเรี่ยราด~~ลดการจอง heap ไปทั่ว

## วิธีลด heap allocation (เท่าที่ผมรู้)
### 1. สร้าง Buffer ให้มีขนาดที่พอใช้งานไว้ล่วงหน้า
การจัดสรรหน่วยความจำแบบไดนามิก (เช่น ในระหว่างการทำงาน) มักจะนำไปสู่การจัดสรรจาก heap ซึ่ง GC จะต้องเรียกคืนในที่สุด แทนที่จะสร้าง slices หรือบัฟเฟอร์ใหม่ในขณะที่ทำงาน การจัดสรรบัฟเฟอร์ที่นำกลับมาใช้ใหม่ล่วงหน้าจะช่วยลดการจัดสรร

#### ตัวอย่าง:
```go
func processData(inputs [][]byte) {
    // สร้าง buffer ล่วงหน้าขนาด 1024 (โดยคาดการจากจำนวนของ slice ย่อยของ inputs)
    buffer := make([]byte, 1024)

    for _, input := range inputs {
        n := copy(buffer, input)
        // ประมวลผล 'buffer' โดยไม่สร้าง slices ใหม่
        fmt.Printf("Processed %d bytes\n", n)
    }
}
```
#### ประโยชน์:

- ลดการจัดสรรจาก heap ที่เกิดขึ้นซ้ำๆ
- บัฟเฟอร์ถูกนำกลับมาใช้ใหม่ ซึ่งช่วยลดภาระงานของ garbage collector

### 2. การใช้ sync.Pool สำหรับการนำกลับมาใช้ใหม่
sync.Pool เป็นเครื่องมือที่มีประสิทธิภาพสำหรับการจัดการวัตถุชั่วคราวที่ใช้บ่อยและถูกทิ้ง มันช่วยบรรเทาค่าใช้จ่ายของการจัดสรรและการเก็บขยะโดยการทำให้วัตถุที่นำกลับมาใช้ใหม่พร้อมใช้งาน

#### ตัวอย่าง:
```go
import (
    "sync"
)

var bufferPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 1024)
    },
}

func handleRequest(data []byte) {
    buf := bufferPool.Get().([]byte) // รับบัฟเฟอร์จากพูล
    defer bufferPool.Put(buf)        // นำกลับไปเพื่อใช้ใหม่

    copy(buf, data)
    // ประมวลผลบัฟเฟอร์...
}
```
#### ประโยชน์:

- ลดจำนวนการจัดสรรอย่างมีนัยสำคัญ
- บัฟเฟอร์ถูกนำกลับมาใช้ใหม่ ซึ่งสามารถลดแรงกดดันจาก GC ได้อย่างมาก

### 3. การจัดการความจุของ Slice
Slices ใน Go มีประโยชน์มาก แต่การเติบโตที่ไม่สามารถควบคุมได้ของ slice มักจะส่งผลให้เกิดการจัดสรรจาก heap โดยการจัดการความจุของ slice อย่างระมัดระวังหรือหลีกเลี่ยงการปรับขนาดที่ไม่จำเป็น คุณสามารถเก็บ slices ไว้ใน stack แทนที่จะเป็น heap

#### ตัวอย่าง: หลีกเลี่ยงการเติบโตของ Slice
```go
func collectData(inputs []int) []int {
    result := make([]int, 0, len(inputs)) // จัดสรรล่วงหน้าด้วยความจุ

    for _, val := range inputs {
        result = append(result, val)
    }

    return result
}
```
#### คำอธิบาย:

โดยการเริ่มต้น result ด้วยความจุที่กำหนดไว้ล่วงหน้า (len(inputs)) คุณจะหลีกเลี่ยงการจัดสรรใหม่และลดโอกาสในการจัดสรรจาก heap ที่เพิ่มขึ้น

### 4. การจัดสรรแบบศูนย์สำหรับสตริง
สตริงใน Go เป็นแบบไม่เปลี่ยนแปลง หมายความว่าการแก้ไขแต่ละครั้งจะสร้างสตริงใหม่ เพื่อหลีกเลี่ยงการจัดสรรสตริงบ่อยๆ:

- ใช้ strings.Builder สำหรับการเชื่อมต่อสตริง
- หลีกเลี่ยงการใช้ + สำหรับการเชื่อมต่อสตริงหลายๆ ตัวในลูป

#### ตัวอย่าง:
```go
import "strings"

func buildMessage(parts []string) string {
    var builder strings.Builder
    builder.Grow(100) // จัดสรรพื้นที่ล่วงหน้า

    for _, part := range parts {
        builder.WriteString(part)
    }

    return builder.String()
}
```
#### ประโยชน์:

- strings.Builder ได้รับการปรับให้เหมาะสมเพื่อการจัดสรรที่น้อยลง
- การเตรียมพื้นที่ล่วงหน้าช่วยหลีกเลี่ยงการจัดสรรใหม่ที่ไม่จำเป็น

### 5. ใช้การจัดสรรใน Stack เมื่อเป็นไปได้
หนึ่งในการเพิ่มประสิทธิภาพที่มีประสิทธิภาพที่สุดของ Go คือการวิเคราะห์การหลบหนี — เทคนิคของคอมไพเลอร์ที่กำหนดว่าตัวแปรใดสามารถจัดสรรได้อย่างปลอดภัยใน stack หรือจำเป็นต้องหลบหนีไปที่ heap

#### ตัวอย่าง: หลีกเลี่ยงการหลบหนีของ Pointer
```go
type Data struct {
    value int
}

func processData() Data {
    d := Data{value: 42}
    return d // คอมไพเลอร์สามารถจัดสรร 'd' ใน stack
}

func processPointer() *Data {
    d := Data{value: 42}
    return &d // 'd' หลบหนีไปที่ heap ทำให้เกิดการจัดสรร
}
```
#### แนวทางการปฏิบัติที่ดีที่สุด:

- หลีกเลี่ยงการส่งคืน pointers ไปยังตัวแปรภายในเว้นแต่จำเป็นต้องทำ
- ให้ความสำคัญกับค่ามากกว่าพอยน์เตอร์เมื่อขนาดของวัตถุมีขนาดเล็ก

### 6. ลดการจัดสรรในเส้นทางที่ร้อน
เส้นทางที่ร้อนคือส่วนของโค้ดที่ถูกดำเนินการบ่อย (เช่น ตัวจัดการคำขอ, การวนลูป) การกำจัดการจัดสรรในส่วนสำคัญเหล่านี้สามารถนำไปสู่การปรับปรุงประสิทธิภาพในระดับใหญ่

#### ตัวอย่าง:
```go
func calculate(inputs []int) int {
    sum := 0
    for _, val := range inputs {
        sum += val // ใช้หน่วยความจำเดิมในการทำงาน
    }
    return sum
}
```
#### คำอธิบาย:

โดยการหลีกเลี่ยงการขยาย slice การจัดสรรจาก heap ที่มากเกินไป และการจัดสรรใหม่ในเส้นทางที่สำคัญต่อประสิทธิภาพ ภาระงานของ GC สามารถลดลงได้


## ข้อผิดพลาดทั่วไปของการเขียนโปรแกรมแบบไม่มีการจัดสรร
- ความชัดเจนกับการแลกเปลี่ยนประสิทธิภาพ: เทคนิคการจัดสรรแบบศูนย์อาจทำให้โค้ดซับซ้อนและอ่านยากมากขึ้น สิ่งสำคัญคือต้องชั่งน้ำหนักประสิทธิภาพที่ได้รับกับการบำรุงรักษา
- ความเสี่ยงจากการจัดการหน่วยความจำด้วยตนเอง: แตกต่างจากภาษาเช่น C/C++ นักพัฒนา Go ไม่คุ้นเคยกับการจัดการหน่วยความจำด้วยตนเอง การใช้พูลหรือบัฟเฟอร์ที่จัดสรรล่วงหน้าอย่างไม่ถูกต้องอาจนำไปสู่ข้อบกพร่องทางตรรกะ เช่น การใช้ข้อมูลหลังจากที่ถูก "ปล่อย" ไปแล้ว
- การวิเคราะห์เป็นกุญแจสำคัญ: สิ่งสำคัญคือต้องวิเคราะห์แอปพลิเคชันของคุณก่อนและหลังการปรับปรุง ใช้เครื่องมือเช่น pprof เพื่อให้แน่ใจว่าความพยายามในการจัดสรรที่เป็นศูนย์ของคุณช่วยปรับปรุงประสิทธิภาพจริงๆ และไม่ทำให้โค้ดยากต่อการบำรุงรักษา

## สรุป
การเขียนโปรแกรมแบบไม่มีการจัดสรรใน Go เป็นเทคนิคขั้นสูงที่ต้องการการวางแผนอย่างรอบคอบและความรู้เชิงลึกเกี่ยวกับการจัดการหน่วยความจำใน Go โดยการใช้เครื่องมือเช่น sync.Pool การจัดสรรบัฟเฟอร์ล่วงหน้า การจัดการความจุของ slice และการใช้การจัดสรรใน stack อย่างมีประสิทธิภาพ คุณสามารถลดแรงกดดันจาก GC อย่างมีนัยสำคัญและบรรลุประสิทธิภาพที่ราบรื่นมากขึ้น

อย่างไรก็ตาม ด้วยพลังอันยิ่งใหญ่จึงมาพร้อมกับความรับผิดชอบ — การปรับปรุงเหล่านี้ควรใช้ด้วยความระมัดระวัง โดยมุ่งเน้นไปที่ส่วนของโค้ดที่สำคัญต่อประสิทธิภาพ เป้าหมายควรเป็นการสร้างสมดุลระหว่างการบรรลุการปรับปรุงประสิทธิภาพและการรักษาความชัดเจนของโค้ด

ยอมรับแนวทางการจัดสรรที่เป็นศูนย์เมื่อจำเป็น และคุณสามารถเขียนโค้ด Go ที่ไม่เพียงแต่มีประสิทธิภาพ แต่ยังสามารถขยายได้สำหรับแอปพลิเคชันในโลกจริงที่ต้องการ
